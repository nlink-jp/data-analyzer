# アーキテクチャ: data-analyzer

## このツールが存在する理由

大規模ログデータ（10万件以上）をLLMで分析する場合、全データをプロンプトに投入するナイーブなアプローチはコンテキストウィンドウの制限に衝突する。一般的な2つの回避策にはそれぞれ根本的な欠陥がある：

1. **切り捨て（Truncation）** — データの一部しか見えない。データセット全体にまたがるパターンは見逃される。
2. **Map-Reduce** — 各チャンクを独立に分析し、結果を統合する。しかしReduce段階で中間コンテキストが過度に圧縮され、チャンク間にまたがる微妙なパターンが失われる。実際の運用で精度劣化を確認済み。

data-analyzerは**スライドウィンドウ＋段階的要約方式**でこの問題を解決し、データセット全体のコンテキストを保持する。

## なぜスライドウィンドウ＋段階的要約なのか

核心的な洞察：人間のアナリストも10万件を一度に読むわけではない。一定量を読み、メンタルモデルを形成し、そのモデルを持って次の一定量を読む。新しい観察がモデルを精緻化・更新する。

スライドウィンドウエンジンはまさにこれを行う：

```
ウィンドウ0: [RAWデータチャンク0] → LLM → 要約₀ + Findings₀
ウィンドウ1: [要約₀] + [Findings₀] + [RAWデータチャンク1] → LLM → 要約₁ + Findings₁
ウィンドウ2: [要約₁] + [Findings₀₊₁] + [RAWデータチャンク2] → LLM → 要約₂ + Findings₂
...
```

**Map-Reduceに対する優位性：**
- ランニングサマリーがコンテキストを運搬する「メンタルモデル」として機能する。Map-Reduceにはこの等価物がない — 各Mapperはステートレス。
- Findingsは要約とは別に蓄積される。これにより、重要な発見が要約の圧縮で失われることを防ぐ。
- ウィンドウのオーバーラップにより、チャンク境界近傍のレコードがコンテキストから切り離されない。

**なぜRAG/埋め込みベースの検索ではないのか？**
- RAGは特定の質問に対して効果的だが、発見指向の分析（「異常を見つけよ」）にはデータを順次処理する必要がある。パターンを見るまで何を検索すべきか分からない。
- 埋め込みベースの類似度検索は時系列パターンを見逃す（例：個々のレコードは正常に見えるが、連続したブルートフォースとしてのみ不審となるもの）。

## なぜウィンドウサイズに上限を設けるのか（max_records_per_window）

トークン予算上は500件以上のレコードが1ウィンドウに収まる場合でも、LLMの出力品質は大量入力で著しく劣化する。実運用で確認された事象：559件のレコードは数学的に128Kトークンに収まるが、モデルはJSONを途中で切断し、citationのフィールドを省略し、少ないウィンドウに比べて浅い分析を生成した。

`max_records_per_window`（デフォルト200）はトークン予算とは無関係にハードキャップを強制する。これは**品質ガード**であり、メモリガードではない。トークン予算は「何が収まるか」を制御し、レコード上限は「何が良い結果を生むか」を制御する。

559件でmax 200の場合、エンジンは200件×約3ウィンドウ（＋オーバーラップ）で処理し、559件の1ウィンドウより優れたFindingsを生成する。

## なぜFindingsを要約から分離するのか

Findingsと要約は異なる目的を持つ：

- **要約**はこれまでに見たすべての圧縮表現。分析が進むにつれ成長後安定化する。新しいデータを解釈するためのコンテキストを提供する。
- **Findings**は個別の、引用可能な発見事項。ハルシネーション防止のため引用（レコードインデックス）を保持し続けなければならない。単調に増加する。

Findingsを要約に折り込むと2つの問題が生じる：
1. 要約の圧縮が引用を剥ぎ取り、主張の検証が不可能になる。
2. 初期の重要なFindingsが要約の進化に伴い希薄化される。

分離することで、優先度ベースのevictionも可能になる：Findingsがコンテキスト予算を超えた場合、低重要度のものをFIFO方式で排出しつつ、高重要度のものは常に保持する。排出されたFindingsは最終出力には残る — 後続のLLMコンテキストに含まれないだけである。

## なぜcitation検証（信頼ではなく検証）なのか

すべてのFindingは`[Record #N]`インデックスで少なくとも1つのソースレコードを参照しなければならない。しかし**LLMが生成したexcerptは決して信頼しない**。代わりにエンジンは3層の検証を行う：

### 第1層：レコードインデックスの妥当性検証
レコードインデックスが範囲外であれば、そのcitationを除去する。LLMがレコード番号自体を捏造したケースを捕捉する。

### 第2層：関連性チェック
LLMが生成したexcerptの値がオリジナルレコードに実際に存在するかを確認する。excerptの非自明な値が少なくとも1つオリジナルJSONに含まれていれば、そのcitationは関連性ありと判定する。これにより捏造された値を捕捉しつつ、フィールドの選択的引用（LLMがフィールドの一部のみを引用）は許容する。

### 第3層：オリジナルへの強制差替
関連性の有無にかかわらず、excerptは**常にオリジナルレコード全体に差し替えられる**。LLMのexcerptはフィールドの省略、値の再フォーマット、部分的なデータの包含が常態化している。オリジナルに差し替えることで、最終レポートのすべてのcitationには完全で未修正のソースデータが含まれる。

### ハルシネーション警告
citationのexcerpt値がオリジナルレコードに全く存在しない場合（第2層が不合格）、エンジンは警告を発する。citationはオリジナルレコード付きで残す — ユーザーは実データを見てFindingが妥当かどうかを判断できる。

### citation復旧
Findingにcitationが全く付いていない場合（LLMが空の`citations[]`を返した場合）、エンジンはFindingのdescriptionテキストから`Record #N`参照を抽出し、オリジナルレコードからcitationを構築する。LLMが散文中では参照したがJSONとして構造化しなかったevidenceを復旧する。

**なぜLLMのexcerptを検証して保持しないのか？**
ローカルLLMは技術的に有効なJSONだが重要なフィールドを省略したexcerptを一貫して生成するため。559件のテストでは、すべてのcitationが修正を必要とした — LLMは常にフィールドのサブセットのみを選択していた。部分的なexcerptを表示すると、アナリストにとって重要なコンテキストが隠蔽される。

## なぜ引用を義務化するのか

すべてのFindingは`[Record #N]`インデックスで少なくとも1つのソースレコードを参照しなければならない。これは意図的なハルシネーション対策である。

ローカルLLMは特にもっともらしい捏造を生成しやすい。引用を義務付けることで検証可能なチェーンが生まれる：compileステップで元のレコードを参照し、Findingがモデルの事前知識ではなく実データに基づいていることを確認できる。

## なぜ128K設計上限なのか

ターゲットモデル（gemma-4-26b-a4b）は256Kトークンのコンテキストウィンドウを持つ。意図的に128Kに制限する理由：

1. **品質は端部で劣化する** — モデルはコンテキスト制限に近づくほど出力品質が低下する。50%のマージンで高品質ゾーンに留まる。
2. **レスポンス予算** — モデルが出力を生成するための余地が必要（5K確保）。
3. **安全マージン** — トークン推定は近似値（後述）のため、バッファがなければ切り捨てエラーが発生する。

## なぜ二重トークン推定なのか

CJK/ASCII/JSON混在コンテンツのトークン推定は本質的に近似である。2つの独立した推定器を使い、大きい方を採用する：

1. **ワードベース** — CJK文字×2 + ASCIIワード×1.3 + 句読点文字数。自然言語の散文に適する。
2. **文字ベース** — 総文字数÷4。構造化データ（JSON）に適する。句読点が支配的な場合に有効。

**なぜ二重方式なのか？** 初期バージョンはワードベース推定のみを使用し、JSONの句読点（`{`, `"`, `:`, `,`）を0トークンとしてカウントしていた。これによりJSONデータで約4-5倍の過小推定が発生し、LLMが実効的に処理できる約200件に対して559件を1ウィンドウに投入してしまった。文字ベース推定器がこれらのケースを捕捉する。

## なぜメモリマップなのか

128K予算は静的ではなく動的に分割する。分析の進行に伴い構成が変化するため：

```
初期ウィンドウ:   [2K system] [0 要約] [0 findings] [121K データ] [5K response]
中間ウィンドウ:   [2K system] [10K 要約] [8K findings] [103K データ] [5K response]
後期ウィンドウ:   [2K system] [15K 要約] [20K findings] [86K データ] [5K response]
```

後半のウィンドウはウィンドウあたりの処理レコード数が減少するが、これは許容範囲である：
- 蓄積された要約とFindingsからのリッチなコンテキストを持つ。
- データセット全体にまたがるパターンは既に前段の要約に捕捉されている。
- オーバーラップにより取りこぼしが発生しない。

注：メモリマップは`max_records_per_window`と併用される。実際のウィンドウサイズは`min(トークン予算が許す件数, max_records_per_window)`。

## なぜ3つのサブコマンドなのか

`prepare → analyze → compile`パイプラインはUnix哲学に従う：各ステップが1つのことを行い、中間成果物を生成し、独立して実行できる。

**なぜ単一コマンドではないのか？**
- `prepare`は対話的、`analyze`はバッチ処理。混合すると使いにくいUXになる。
- パラメータファイルは再利用可能：同じ分析を異なるデータセットに実行したり、LLM会話を再実行せずにパラメータを調整できる。
- `compile`は再分析なしで再実行可能：レポート形式の変更、セクションの追加、他のツールへのパイプが可能。
- デバッグが容易：`params.json`と`result.json`を独立して検査できる。

**なぜ`prepare --input`なのか？**
ユーザーが大きな複数行プロンプトをターミナルにペーストした場合、`bufio.Scanner`は最初の1行のみを読み取り、残りの行がシェルに漏れてコマンドとして実行される。実際にインシデントが発生した（zshがドメイン名をコマンドとして実行）。`--input`はファイルから要件を読み込むことでターミナルペーストの問題を完全に回避し、その後対話的リファインモードに入る。

## なぜ出力言語制御なのか

LLMはプロンプトの言語（この場合英語）でデフォルト応答する。日本語のログデータをレビューする日本語話者のアナリストにとって、英語レポートは不要な摩擦を生む。`--lang`フラグはウィンドウ分析プロンプトと最終レポートプロンプトの両方に言語指示を注入し、要約テキストとFindingのdescriptionを指定言語で生成することを保証する。

言語は3レベルで設定可能（優先度が高い順）：
1. CLIフラグ: `--lang Japanese`
2. パラメータファイル: params.jsonの`"lang": "Japanese"`
3. 設定: config.tomlの`analysis.lang`または環境変数`DATA_ANALYZER_LANG`

## なぜパース失敗時にリトライするのか

ローカルLLMは`nlk/jsonfix`で修復できない不正なJSONを時折生成する（例：markdownフェンスの途中で文字列が切断）。ウィンドウ全体の分析を破棄するのではなく、エンジンは1回リトライする。LLM出力は非決定的なため、同じプロンプトでも2回目は有効なJSONを生成することが多い。

2回とも失敗した場合、そのウィンドウをスキップして分析を継続する。ジョブ全体を中断するより望ましい — 先行する数時間の計算を無駄にしないため。

## なぜチェックポイントと冪等性なのか

10万件の分析にはかなりの時間（ローカルLLMでは数時間）を要する。2つの保証がこれを実用的にする：

1. **チェックポイント** — 各ウィンドウ処理後にエンジンが状態（要約、Findings、レコードオフセット）を保存する。中断された場合（Ctrl+C、クラッシュ、電源喪失）、最後のチェックポイントから分析を再開する。これがなければ、いかなる中断も先行する全計算を無駄にする。

2. **冪等性** — 完了したジョブはキャッシュされる。同じ分析を再実行するとキャッシュ結果が即座に返される。これにより誤った再実行が防止され、スクリプトや自動化での使用が安全になる。

チェックポイントはアトミックファイル書き込み（一時ファイルに書き込み後rename）を使用し、書き込み中のクラッシュによる破損を防止する。

## なぜOpenAI互換API（SDKなし）なのか

LLMクライアントはSDKではなく生HTTPを使用する理由：

1. **外部依存なし** — OpenAI Go SDKは推移的依存関係とバージョン変更を持ち込む。必要なAPI表面は極めて小さい（1エンドポイント）。
2. **互換性** — LM Studio、Ollama、vLLMなどのローカルサーバーはすべて同じ`/chat/completions`エンドポイントを実装している。薄いHTTPクライアントの方が、意見の強いSDKよりもこれらの差異を柔軟に処理できる。
3. **テスト容易性** — `httptest.Server`により、複雑なSDKインターフェースをモックすることなくクライアントを容易にテストできる。

クライアントはインターフェース（`llm.Client`）の背後にあるため、エンジンはモックでテスト可能であり、コアアルゴリズムを変更せずに代替バックエンドを追加できる。

## なぜnlk統合なのか

`nlk`ライブラリはローカルLLMとの対話に不可欠な5つのユーティリティを提供する：

| モジュール | 理由 |
|-----------|------|
| `guard` | 非信頼データをナンス付きXMLでラップしプロンプトインジェクションを防止。ログデータには命令に見えるテキストを含む可能性がある。|
| `strip` | `<think>`タグを除去。ローカルモデル（特にGemma）はJSON解析前に除去すべき思考過程推論を出力する。|
| `jsonfix` | 不正なJSONを修復。ローカルモデルは末尾カンマ、引用符欠落、markdownフェンスなどのJSON不備を頻繁に生成する。|
| `validate` | ルールベースの出力検証。severity値が許可されたセットに属することを保証する。|
| `backoff` | 指数リトライ。ローカルLLMはメモリ圧力や並行負荷下で時折失敗する。|

これらは複数のnlink-jpプロジェクト（mail-analyzer-local、gem-query、lite-llm）で実戦テスト済みであり、ローカルLLMの全範囲の癖に対応する。

## なぜGoなのか

- **単一バイナリ配布** — macOSとWindowsでランタイム依存なしに動作する必要がある。Goの静的コンパイルが最適。
- **エコシステムの一貫性** — util-seriesの既存ツールの大半がGo。共有パターン（Cobra CLI、TOML設定、nlkライブラリ）が認知的オーバーヘッドを削減する。
- **CGO不要** — gem-query（DuckDB）と異なりC依存がないため、5つの全ターゲットプラットフォームでクロスコンパイルが機能する。
